# Tiled/ — Self-Contained Map Sandbox

A reference `.tmj` map built to the exact schema the project's Tiled pipeline
(`src/io/tiledLoader.ts` → `src/io/mapDataBridge.ts` → `src/rendering/tiledTilemapRenderer.ts`)
expects. Kept separate from `public/maps/` until a future milestone wires it
into a live Scenario. Everything in this directory is self-contained — the
`.tmj` references `temple_ruins.png` by a local relative path, so the whole
directory can be copied or moved as a unit.

## Files

| File | What it is |
|---|---|
| `temple_courtyard.tmj` | 16×12 Tiled map, embedded tileset, 6 layers |
| `temple_ruins.png` | 256×96 px tileset, 24 tiles @ 32px, zero spacing. Renders at 2× upscale via the container's `TILE_SIZE / tilewidth` ratio; no `filter` property set on the tileset so it defaults to `GL_NEAREST` (correct for pixel art). Generated by ↓ |
| `gen_tileset.mjs` | Pure-Node PNG encoder — regenerates the tileset |
| `validate.mjs` | Runs the `.tmj` through `mapDataBridge.ts`, 47 assertions |
| `README.md` | This file |

## Commands

```bash
node Tiled/gen_tileset.mjs      # regenerate temple_ruins.png
npx tsx Tiled/validate.mjs      # validate .tmj loads cleanly (exit 0 on pass)
```

## Tile palette

Local ID → GID is `id + 1` (firstgid is 1). Only tiles with game-mechanical
properties get a `tiles[]` entry in the tileset; the rest are visual-only.

The tileset PNG overlays a **glyph** on mechanically-meaningful tiles so you
can read terrain semantics directly from the rendered map:

| Glyph | Meaning |
|---|---|
| ╱╱ diagonal hatch | difficult terrain (`moveCost ≥ 2`) |
| □ hollow square | standard cover (`coverGrade: 1`) |
| ⊡ nested squares | greater cover (`coverGrade: 2`) |
| ✕ cross | impassable (`blocked: true`) |
| ∧ chevron | raised (`elevation ≥ 1`) |

## Layer stack

Array order in `layers[]` is bottom-to-top (Tiled's draw order). The bridge
reads tile properties from all visible tile layers — higher layers override
lower ones for numeric properties (`moveCost`, `coverGrade`, `elevation`), and
`blocked` is additive (any layer can mark a tile blocked).

| # | Name | Type | Contents |
|---|---|---|---|
| 1 | `Ground` | tilelayer | 192 cells filled — flagstone base, moss ring, water under pool |
| 2 | `Terrain` | tilelayer | Sparse overlay — carries all `moveCost`/`coverGrade`/`elevation` |
| 3 | `Walls` | tilelayer | Perimeter walls + entrance gap, all `blocked` |
| 4 | `Spawns` | objectgroup | 6 spawn objects, `type:"spawn"`, pixel coords |
| 5 | `Hazards` | objectgroup | 1 rectangle, `type:"hazard"`, covers the pool |
| 6 | `Objectives` | objectgroup | 1 point, `type:"defeat_all_enemies"` |

## Live in the picker

A migrated copy is served at `public/maps/temple_courtyard.tmj` (with
`public/tilesets/temple_ruins.png`) as the 9th entry in `ScenarioLoader.tsx` —
the first picker entry to use the direct-`.tmj` load branch
(`scenarioLoader.ts:862`). The eight prior entries all use JSON wrappers with
a `tiled_map` reference (hybrid path — `.tmj` for terrain only).

Manual browser test confirmed: map renders, PCs and enemies both route around
pillars/statue/walls (Dijkstra respects the `blocked` set extracted by the
bridge), and the cursed-pool hazard zone is wired into `battleMap.hazards`
for `tickHazardZones` on `end_turn`.

The `Tiled/` originals stay self-contained — the copy's only diff is
`"image": "../tilesets/temple_ruins.png"`. To re-migrate after editing here:

1. `cp Tiled/temple_courtyard.tmj public/maps/`
2. `cp Tiled/temple_ruins.png public/tilesets/`
3. In the copied `.tmj`, change `"image": "temple_ruins.png"` →
   `"image": "../tilesets/temple_ruins.png"` (paths are relative to the map
   file — `tiledLoader.ts:resolveUrl` handles the `..`).
